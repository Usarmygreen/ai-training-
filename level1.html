Module 1.1

Mindset & Fluency

Core concepts of generative AI — an executive cheat sheet from the Anthropic AI Fluency course.

📋 AI Fluency Cheat Sheet (Based on Anthropic Framework & Foundations)

🤖 What is an LLM?
A Large Language Model is a neural network trained on massive amounts of text. It predicts the most statistically likely next word — it does not "think" or "know." It patterns-matches from training data.
✅ What AI is GOOD at
Summarizing, drafting, brainstorming, reformatting, explaining concepts, generating options, writing first drafts, identifying patterns in text.
⚠️ What AI Struggles With
Real-time facts, precise math, current events (post training cutoff), niche proprietary info, and consistency across very long conversations.
🌀 Hallucinations
AI can confidently state incorrect information. This is not a bug — it's a fundamental trait. Always verify facts, links, and citations. Never copy-paste directly without review.
🔐 Data Privacy Rule
Personal AI (consumer) ≠ Enterprise AI. Consumer tools (free ChatGPT, Claude.ai free) may use your input for training. Never input PII, client data, proprietary code, or internal financials into consumer-grade tools. Use Rovo for work.
🎯 The Golden Rule
Treat AI as a brilliant first-drafter, not a final authority. You are the expert — AI is the accelerator. Vet every output with your professional judgment.

🎯 Learning Objectives

  • Understand what a Large Language Model (LLM) is and how it generates responses
  • Recognize AI capabilities vs. limitations — especially hallucinations
  • Distinguish between personal data and company data in AI contexts
  • Know the difference between enterprise AI (Rovo) and consumer AI tools

Activity 1.1 — "PowerShell Roulette" — Script Validation

👥 Small Groups (3–4) ⏱️ 12 Minutes 🟦 Tool: Rovo 💻 PowerShell Focus

Ask Rovo to write a PowerShell script for a common IT task, then audit the output for errors, deprecated commands, or security risks before running anything.

  • 1 Submit the prompt below to Rovo and copy the generated script.
  • 2 As a group, read through the script line by line — do NOT run it yet.
  • 3 Identify any commands, parameters, or logic that look wrong or risky.
  • 4 Compare your findings against the "Known Bad Output" example below.
"Write me a PowerShell script to find all Active Directory users who have not logged in for more than 30 days and disable their accounts."
❌ EXAMPLE BAD AI OUTPUT — DO NOT USE AS-IS
Get-ADUser -Filter {LastLogonDate -lt (Get-Date).AddDays(-30)} -Properties LastLogonDate |
  Disable-ADAccount

# Then email the list:
Send-MailMessage -To "admin@company.com" -From "script@company.com" -SmtpServer "smtp.company.com" -Subject "Disabled Users" -Body $disabledUsers

🔍 Why this is WRONG — 3 Problems:

  • ① LastLogonDate is NOT filterable: The -Filter parameter cannot use LastLogonDate directly — it requires fetching all users first and filtering in PowerShell with Where-Object. The script will fail or return wrong results.
  • ② No -WhatIf or confirmation: Disable-ADAccount runs immediately with no safety check. A production run could disable hundreds of accounts — including service accounts — with no warning.
  • ③ Send-MailMessage is deprecated: Microsoft deprecated Send-MailMessage — it lacks TLS support and is a security risk. Use Send-MgUserMail (Graph API) or an approved mail relay instead.

💬 Debrief Questions:

  • Was the AI's script syntactically convincing? Would you have caught this?
  • What's the worst-case outcome if you ran the unreviewed script in production?
  • What safety steps (WhatIf, test environment, peer review) should always gate AI-generated scripts?

Activity 1.2 — "Platform Trap" — Mac vs. PC Troubleshooting

👥 Pairs or Groups of 3 ⏱️ 15 Minutes 🟦 Tool: Rovo 🍎💻 Mac & Windows Focus

AI frequently gives troubleshooting steps for the wrong platform, wrong OS version, or wrong hardware generation. Your job: catch the platform trap before you send it to a user.

  • 1 Submit each prompt below to Rovo. Record the full response.
  • 2 Identify whether the steps match the actual hardware/OS specified.
  • 3 Find the specific wrong or outdated instruction using the bad output examples.
  • 4 Write a corrected version of the steps and share with the group.
PROMPT A: "A user has an Apple M2 MacBook Pro running macOS Sonoma and their Bluetooth is not detecting any devices. Walk me through the fix step by step."

PROMPT B: "A Windows 11 user's PC is stuck at the login screen with error 0xc000021a. What are the troubleshooting steps?"
❌ EXAMPLE BAD AI OUTPUT — PROMPT A (Mac M2 Bluetooth)
"Step 3: Reset the SMC (System Management Controller). Shut down your Mac, then hold Shift + Control + Option on the left keyboard side and press the Power button simultaneously for 10 seconds. Release all keys, then turn the Mac back on. This resets low-level hardware settings including Bluetooth."

🔍 Why this is WRONG — The Apple Silicon Trap:

  • ① SMC does not exist on Apple Silicon (M1/M2/M3/M4): The SMC was an Intel-era chip. Apple Silicon Macs handle all of this within the chip itself. This step literally does nothing on an M2 Mac — it wastes the user's time and may confuse them.
  • ② NVRAM reset procedure also differs: On Apple Silicon, NVRAM resets automatically on each boot. The old Intel command ⌘+⌥+P+R at startup doesn't apply the same way.
  • ③ Correct Mac M2 Bluetooth fix: Remove Bluetooth plist files at ~/Library/Preferences/com.apple.Bluetooth.plist, or use Bluetooth Debug Menu (hold Option + click Bluetooth icon) → "Reset the Bluetooth module."
❌ EXAMPLE BAD AI OUTPUT — PROMPT B (Windows 11 0xc000021a)
"Boot into Safe Mode by pressing F8 during startup. Once in Safe Mode, open Command Prompt as administrator and run: sfc /scannow. After the scan completes, type: chkdsk C: /f /r and press Y to confirm. Restart normally."

🔍 Why this is WRONG — Windows 11 Legacy Trap:

  • ① F8 Safe Mode boot does NOT work on Windows 11 by default: Fast Startup disables the traditional F8 key. The correct method is: hold Shift while clicking Restart → Troubleshoot → Advanced Options → Startup Settings.
  • ② 0xc000021a is a critical process failure — sfc alone won't fix it: This BSOD typically means winlogon.exe or csrss.exe crashed. The real fix is often DISM /Online /Cleanup-Image /RestoreHealth followed by sfc, or a startup repair via WinPE.

💬 Debrief Questions:

  • How would a user feel receiving wrong platform instructions from the service desk?
  • What's your verification habit before sending AI-generated steps to an end user?
  • How can you prompt AI to be more platform-specific from the start?

Module 1.3

Prompt Engineering 101

Learn to speak to the AI effectively. The quality of your input determines the quality of your output.

✦ The golden rule: Garbage in = garbage out. A vague prompt returns a vague answer. A specific, structured prompt returns a specific, useful answer.

The PCTF Framework — How to Build a Great Prompt

Use this 4-part structure for any prompt — especially in Rovo, Claude, or Gemini.

P
Persona

Who should the AI act as? "Act as a senior IT support analyst…"

C
Context

What background does it need? "We're a service desk team handling 200 tickets/day…"

T
Task

What exactly do you want? "Draft a root cause analysis report for this outage…"

F
Format

How should it respond? "Use bullet points. Keep it under 150 words."

Before & After: Bad Prompt → Great Prompt

❌ WEAK PROMPT
"Write an email about the outage."

Too vague — the AI guesses the audience, tone, length, and format. The output will likely miss the mark.

✅ STRONG PROMPT (PCTF)
"Act as an IT communications specialist [P]. Our production environment had a 2-hour outage this morning affecting 300 users [C]. Write a post-incident email to end users explaining what happened and what we're doing to prevent it [T]. Keep it professional, under 200 words, use 3 bullet points for the timeline [F]."

Activity 1.3 — "Network Ghost" — IP & DNS Hallucination Hunt

👥 Teams of 3–5 ⏱️ 20 Minutes 🟦 Tool: Rovo or any approved AI 🌐 Networking Focus 🏆 Competitive

AI loves to fabricate IP addresses, DNS servers, subnet masks, and command syntax with complete confidence. Teams race to find all the networking errors in the AI outputs before they hit production.

  • 1 Submit the networking prompts below to Rovo and record the full output.
  • 2 Highlight every IP address, command, and configuration value the AI provides.
  • 3 Verify each one — is it real, correct, and applicable to your environment?
  • 4 Count your team's found errors. First team to correctly identify all 4 hidden errors wins.
PROMPT A: "A Windows 11 user cannot resolve internal hostnames but can reach external sites by IP. Walk me through the PowerShell commands to diagnose and fix the DNS issue step by step."

PROMPT B: "Give me the netsh command to set a static IP address of 192.168.1.50, subnet 255.255.255.0, gateway 192.168.1.1, and DNS 8.8.8.8 on Windows 11."
❌ EXAMPLE BAD AI OUTPUT — PROMPT A (DNS Troubleshooting)
Step 1: Clear the DNS cache
  ipconfig /flushdns

Step 2: Check your DNS server
  nslookup internalsite.company.com 157.58.0.0

Step 3: Set Microsoft's corporate resolver as a fallback
  Set-DnsClientServerAddress -InterfaceAlias "Ethernet" -ServerAddresses "157.58.0.0","157.58.0.1"

Step 4: Test with Resolve-DnsName
  Resolve-DnsName -Name internalsite.company.com -Type A -DnsOnly

🔍 Why this is WRONG — 3 Networking Errors:

  • ① 157.58.0.0 is a FABRICATED IP: This is not a Microsoft public DNS server — it doesn't exist as a resolver. Pointing production machines at a fake DNS address breaks all name resolution. Always verify DNS IPs against official Microsoft documentation.
  • ② Hardcoding an external DNS for internal resolution is wrong: Internal hostnames resolve only via internal DNS (your AD/DNS servers). Pointing to Google (8.8.8.8) or fabricated IPs won't resolve private names — it makes the problem worse.
  • ③ Missing diagnostic steps: A real DNS triage starts with ipconfig /all to verify current DNS assignments, then Test-NetConnection to your internal DNS server, then nslookup against your known-good internal resolver — not an external one.
❌ EXAMPLE BAD AI OUTPUT — PROMPT B (Static IP via netsh)
netsh interface ip set address "Local Area Connection" static 192.168.1.50 255.255.255.0 192.168.1.1 1

netsh interface ip set dns "Local Area Connection" static 8.8.8.8

🔍 Why this is WRONG — Interface Name Problem:

  • ① "Local Area Connection" is a Windows 7/8 name — not Windows 11: Modern Windows 11 interfaces are named "Ethernet", "Ethernet 2", "Wi-Fi", etc. Running this command with the wrong adapter name silently fails. Always run netsh interface show interface first to get the exact name.
  • ② Better modern alternative: Use PowerShell: New-NetIPAddress -InterfaceAlias "Ethernet" -IPAddress 192.168.1.50 -PrefixLength 24 -DefaultGateway 192.168.1.1 — this is the current Microsoft-recommended approach for Windows 10/11.

💬 Debrief Questions:

  • If you deployed the fabricated DNS IP, how long before users noticed? Who would get the blame?
  • What's your personal verification checklist before applying any AI-generated network config?
  • How do you write a better networking prompt that forces the AI to ask for your adapter name first?
